## 第一眼目标

玩家进入经营页 3 秒内，应该不读说明也能看懂：

1. 这是一家怪兽便利店，正在营业。
2. 上面两位怪兽是等待服务的顾客。
3. 高亮顾客是当前服务对象。
4. 订单气泡告诉玩家要拿什么商品。
5. 底部商品卡可以点击取货。
6. 中间托盘是组装和交付位置。
7. 微波炉用于处理带火焰需求的商品。
8. 成功交付会收钱，并让顶部收入变化。

如果这 8 件事需要依赖长文字、Toast 或反复试错才明白，经营页视觉就没有达标。

## 区域差距

| 区域 | 当前问题 | 目标状态 | P0处理方式 | 商业化价值 |
| --- | --- | --- | --- | --- |
| 整体构图 | 功能可运行，但空间关系仍弱 | 店铺、顾客、工作台、商品区一眼分层 | 先不重画背景，改 UI 层级和工作台状态 | 截图更像完整游戏 |
| 顶部 HUD | 数值可读，但图标和权重不够统一 | 时间、收入、满意度/连击、暂停形成稳定信息条 | 复用 `ui_formal_v2/hud`，降低文字占比 | 玩家立刻知道本轮压力和收益 |
| 顾客区 | 顾客可爱，但当前服务对象和订单压力不够强 | 当前顾客、非当前顾客、急躁顾客明确区分 | 强化高亮光圈、订单气泡、耐心条 | 减少玩家误服务和迷路 |
| 订单气泡 | 商品状态和订单完成感偏弱 | 缺失、已入托盘、加热中、完成都可视化 | 做 selected/normal/warning 三套气泡状态 | 玩家知道下一步该点哪里 |
| 工作台 | 微波炉、托盘、收银台像独立物件 | 三者组成同一条服务流程 | 统一位置、阴影、发光和反馈节奏 | 让交付流程更像核心卖点 |
| 托盘 | READY 已有，但存在感不足 | 托盘是整单交付按钮，状态强烈 | 空、部分、READY、阻塞、交付五状态 | 防止玩家以为点商品就能卖出 |
| 商品卡 | 图标不错，但卡片像信息面板 | 商品图标最大，状态靠角标和边框 | 做普通、当前需要、热食、售罄、错误状态 | 提高操作速度和手机可读性 |
| 反馈 | 收钱和错误反馈能跑，但爽感不够 | 成功、Perfect、错误、低耐心有短促动效 | 补金币、评价牌、错误提示底板 | 让玩家愿意继续玩下一单 |
| 文字密度 | 常驻中文标签较多 | 文字降级为辅助，图标和状态优先 | 保留必要数字，删减解释型标签 | 更接近商业游戏截图 |

## P0 美术/UX任务

### A1：HUD 目标统一

解决问题：

- 顶部信息可读，但还不像商业化游戏 HUD。
- 时间、营业额、满意度/连击的优先级需要更明确。

要做什么：

- 以 `assets/resources/ui_formal_v2/hud/` 为第一参考，确定一条正式 HUD。
- 时间居中或最突出，收入与收银反馈关联。
- 暂停按钮改为图标按钮，不占用太多文字空间。

产出：

- HUD 布局目标说明。
- 后续组件目标图：`docs/art/targets/hud-target-v1.png`。

验收标准：

- 手机截图中，时间和收入能在一眼内读出来。
- 顶部不出现密集解释文字。
- HUD 风格与订单气泡、商品卡同属一套粗描边卡通 UI。

### A2：订单气泡状态板

解决问题：

- 当前订单气泡还不足以承担经营页核心信息。
- 玩家需要清楚知道哪些商品缺失、哪些已在托盘、哪些正在加工。

要做什么：

- 固定三种订单气泡：当前顾客、非当前顾客、警告/低耐心。
- 每个订单项至少支持 `missing`、`on_tray`、`processing`、`delivered`、`error`。
- 订单完成后使用盖章、勾选或短促弹跳，而不是只变文字。

产出：

- 订单气泡状态目标说明。
- 后续组件目标图：`docs/art/targets/order-bubble-state-board-v1.png`。

验收标准：

- 不读 Toast 也能知道当前订单还缺什么。
- 当前顾客和非当前顾客的订单气泡差异明显。
- 低耐心时，订单气泡和耐心条能共同制造紧张感。

### A3：顾客舞台与选择高亮

解决问题：

- 双顾客是卖点，但当前画面里“正在服务谁”还可以更强。

要做什么：

- 当前顾客脚下或背后加入高亮光圈。
- 非当前顾客保持可读，但降低视觉抢夺。
- 顾客等待、急躁、开心、生气状态与订单气泡联动。

产出：

- 顾客舞台状态说明。
- 后续组件目标图：`docs/art/targets/customer-stage-target-v1.png`。

验收标准：

- 3 秒内能看出当前服务顾客。
- 切换顾客后，高亮变化明确且不遮挡怪兽。
- 急躁状态不只改耐心条，也能影响顾客表情或气泡。

### A4：工作台统一

解决问题：

- 微波炉、托盘、收银台目前都能用，但还没有形成一条清楚的服务动线。

要做什么：

- 将中部定义为“工作台层”：左/中/右或上/下顺序固定。
- 微波炉处理热食，托盘组装订单，收银台表现收款。
- 三者使用统一阴影、描边、发光强度和弹跳节奏。

产出：

- 工作台区域布局说明。
- 后续组件目标图：`docs/art/targets/workstation-target-v1.png`。

验收标准：

- 商品从商品卡进入订单气泡或微波炉的路径清楚。
- 微波炉 READY 后，玩家知道要点微波炉取出。
- 点击 READY 订单气泡后，玩家知道这是交付动作。

### A5：订单气泡五状态

解决问题：

- 订单气泡是当前玩法的核心准备和交付入口，但视觉状态还需要更清楚。

要做什么：

- 固定订单气泡五状态：`empty`、`partial`、`ready`、`blocked`、`delivered`。
- READY 状态必须有发光、箭头、勾选或明显呼吸动效。
- blocked 状态用于混单、点错顾客、订单已锁定时的红闪/震动。

产出：

- 订单气泡状态规则说明。
- 后续 final art 候选：`assets/ui/final-candidates/gameplay-first-batch-v1/`。

验收标准：

- 订单气泡为空、部分完成、可交付三种状态在截图中明显不同。
- READY 订单气泡像一个可以点击的主要按钮。
- 玩家不会误以为商品点击后已经直接卖出。

### A6：商品卡状态语言

解决问题：

- 当前商品卡可点，但库存、需要、加热、错误等状态还依赖较多文字。

要做什么：

- 固定商品卡状态：普通、当前订单需要、需要加热、售罄/锁定、错误。
- 商品图标保持最大，状态用边框、角标、库存胶囊表达。
- “货架 3”等文字降级，避免抢商品图标焦点。

产出：

- 商品卡状态说明。
- 后续组件目标图：`docs/art/targets/product-card-state-board-v1.png`。

验收标准：

- 玩家能一眼找到当前订单需要的商品。
- 热食商品有统一火焰角标。
- 售罄或错误状态不需要读长提示也能理解。

### A7：反馈与奖励视觉

解决问题：

- 经营页现在完成订单后能收钱，但情绪反馈和奖励爽感偏弱。

要做什么：

- 成功交付：商品飞向顾客或订单气泡，顾客开心，收银台闪光。
- 收款：金币或收入飘字从收银台/顾客方向飞向 HUD。
- 错误：错误商品卡、订单气泡或托盘短暂红闪，不用大段提示遮屏。

产出：

- 反馈状态说明。
- 后续组件目标图：`docs/art/targets/feedback-badge-board-v1.png`。

验收标准：

- 成功交付能带来明确正反馈。
- 错误反馈短促、清楚，不打断下一次操作。
- 收入变化和收银台/顶部 HUD 有视觉连接。

## P0资源结论

P0 不优先重画：

- 店铺背景。
- 怪兽角色本体。
- 商品 icon。
- 微波炉、收银机、托盘基础图。
- 三顾客同屏。
- 装修、广告、IAP、排行榜等外层商业化系统。

P0 优先补齐：

| 优先级 | 资源/组件 | 目的 | 复用基础 |
| --- | --- | --- | --- |
| P0 | 订单气泡 selected/normal/warning | 看懂顾客和订单 | `ui_formal_v2/order` |
| P0 | 订单 item 槽、勾选、错误态 | 看懂缺什么和完成什么 | 代码绘制 + 新 icon |
| P0 | 顾客选择高亮 | 看懂正在服务谁 | 程序动画 + 半透明图 |
| P0 | 订单气泡 READY / partial / done 状态 | 看懂组装和交付 | 新 final art 候选，旧托盘不再作为目标 |
| P0 | 商品卡状态 | 看懂可点、需要、热食、售罄 | 代码绘制 + 角标 |
| P0 | 耐心条和低耐心提示 | 制造时间压力 | 代码缩放 + 颜色 |
| P0 | 金币/收入反馈 | 强化完成订单收益 | 单张 coin + Tween |
| P0 | Good/Perfect/Error 反馈牌 | 增加服务评价感 | 九宫格或代码绘制 |

## 给产品/玩法的反馈

产品规格已经明确了经营页的核心闭环。美术/UX 这边建议产品下一步不要先扩三顾客、多托盘或复杂装修，而是先把前 7 天节奏和订单类型压实：

- 第 1 天：普通单品订单，验证点击商品、订单气泡 READY、交付收钱。
- 第 2-3 天：加入双商品订单，验证订单气泡 partial 和缺失状态。
- 第 3-4 天：加入热食订单，验证微波炉和等待调度。
- 第 5-7 天：加入耐心压力、缺货和更高奖励，验证经营紧张感。

## 给开发/测试的反馈

后续代码改造不应该先大重构，而应该围绕视觉目标做小步落地：

1. 先确认当前 UI 节点结构能否稳定承载 HUD、顾客区、工作台、商品区四层。
2. 再接入订单气泡五状态与订单 item 状态。
3. 然后接商品卡状态、反馈飘字和微波炉 READY 提示。
4. 每一步都用手机截图验收，避免只在编辑器里看起来可用。

## 验收标准

A-001 通过标准：

- 文档同时展示目标图和当前运行图。
- 明确列出 HUD、顾客区、订单气泡、工作台、托盘、商品卡、反馈七类差距。
- 每个 P0 任务都说明了要解决什么、产出什么、怎么验收。
- P0 明确不重画背景、怪兽、商品 icon，避免范围失控。
- 后续能直接进入 `A-002：P0资源清单` 或 `D-001：经营页代码与改造路径审计`。

商业化截图验收标准：

- 在 `390 x 844` 手机尺寸下，不遮挡、不重叠、主要按钮可点击。
- 第一眼能看出当前顾客、所需商品、托盘状态、倒计时和收入。
- 成功交付时，顾客、托盘、收银台、HUD 至少三个位置产生联动反馈。
- 错误操作时，玩家知道错在哪里，并能立刻继续下一步。

## 当前结论

经营页离商业化目标最近的路径不是立刻扩系统，而是先让 V2 目标图对应的闭环“看起来像核心玩法”：

```text
双顾客等待
订单气泡提出需求
商品卡选择商品
微波炉处理热食
订单气泡填充并 READY
点击订单气泡或当前顾客交付
顾客反馈并收钱
```

下一步建议由美术/UX 做 `A-002：P0资源与动效清单`，按 V2 目标图把全量重做资源、临时复用项、P0 动效和高级动效分层列出来。
