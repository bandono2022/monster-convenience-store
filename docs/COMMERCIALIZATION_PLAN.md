# 怪兽便利店商业化推进总计划

更新时间：2026-06-25

## 目标定义

本项目的目标不是只做一个能跑的原型，而是把《怪兽便利店》推进到可以商业化上线的竖屏休闲经营游戏。

商业化上线至少意味着：

- 首屏经营页有明确吸引力，截图能让玩家理解“怪兽便利店”的卖点。
- 新玩家 1 分钟内能完成第一单，并理解订单、商品、托盘、收钱的关系。
- 核心循环能支撑至少 30 至 60 分钟游玩，而不是只完成一轮测试。
- 美术资源风格统一，经营页 UI、顾客、商品、设备不再像不同阶段拼在一起。
- 数值有目标、有压力、有成长，不依赖随手填的价格和耐心。
- 移动端稳定，按钮、文字、商品、订单在手机尺寸下清楚可点。
- 本地版本可以打包给外部玩家试玩，并能收集反馈。

## 当前判断

项目已经具备一个可运行的垂直切片基础：

- 经营中心、经营前准备、营业、结算、升级的基础流程已存在。
- 双顾客、商品点击、微波炉、托盘交付、库存、收入等核心节点已能跑通。
- 文档中已有玩法、M2 经营循环、美术生产计划和验收清单。

但距离商业化上线还有明显差距：

- 经营页是当前最大重点，画面吸引力、信息层级、操作反馈仍不足。
- 美术资源已有一部分，但资源状态、命名、接入优先级还没有形成上线级管线。
- 玩法系统还不完整，顾客、商品、设备、升级、订单压力之间的关系需要收束。
- 数值还没有形成可测试曲线，前 7 天、前 30 分钟体验目标不够明确。
- 测试还偏“能不能跑”，没有形成商业化体验验收。

## 工作方式

默认只在本地推进：

- 不创建 GitHub issue。
- 不推送 GitHub。
- 不提交 git commit。
- 只有用户明确说“同步到 GitHub”“提交”“推送”时才执行。

每个角色开工前，必须先在对话里说明：

```text
【当前角色】
【这次要做什么】
【解决什么问题】
【为什么更接近商业化上线目标】
【产出物是什么】
【验收标准是什么】
```

统筹负责控制顺序。产品、美术、开发/测试不能各做各的，必须围绕当前阶段目标推进。

## 目标图规则

美术和 UI 目标不能只用文字描述，必须通过目标图确认。

后续美术/UX 角色开工时，至少要提供以下其中一种可查看产出：

- 目标整屏图：展示经营页最终希望接近的画面。
- 当前图对比：当前运行截图与目标图并排说明差距。
- UI 组件目标图：HUD、订单气泡、商品卡、托盘、设备、结算反馈等组件的目标样式。
- 资源目标图：角色、商品、设备、背景的资源风格样张。

当前已有经营页基准图：

- `assets/ui/mockups/gameplay-main-order-bubble-ready-v2.png`

后续目标图建议放在：

- `docs/art/targets/`
- `assets/reference/ui_samples/`
- `assets/ui/mockups/`

任何“美术完成”或“UI 方向确认”都必须能用图片判断，不能只依赖文字说明。

## 角色职责

### 统筹 / 制作人

职责：

- 决定当前阶段目标和优先级。
- 拆分产品、美术、开发/测试任务。
- 控制范围，避免把原型变成无限加功能。
- 判断一个任务是否真的更接近上线。

主要产出：

- `docs/COMMERCIALIZATION_PLAN.md`
- `docs/LOCAL_TASK_BOARD.md`
- 阶段验收结论

### 产品 / 玩法策划

职责：

- 定义游戏为什么好玩。
- 把核心循环、订单压力、库存压力、升级动力讲清楚。
- 设计前 7 天和前 30 分钟体验。
- 给开发和测试明确规则，而不是模糊方向。

必须完成：

- 经营页完整玩法流程。
- 商品、顾客、设备、升级、订单生成规则。
- 价格、成本、耐心、奖励、失败惩罚的数值 V1。
- 新手引导与第一轮体验。

### 美术 / UX

职责：

- 定义游戏看起来是否像能上线。
- 统一经营页视觉层级。
- 把“哪些资源能复用、哪些必须重做、哪些先用程序绘制”说清楚。
- 保证操作目标、反馈和商业截图可读。
- 用目标图展示美术和 UI 方向，并用当前图对比说明差距。

必须完成：

- 经营页目标画面与视觉差距清单。
- HUD、顾客区、工作台、托盘、商品卡、结算反馈的规范。
- 正式资源清单、尺寸、命名、导出要求。
- 首批必须替换或补齐的 P0 资源。
- 至少一张经营页目标图或目标图板。

### 开发 / 测试

职责：

- 按产品和美术的已确认规格实现。
- 不自行扩展玩法范围。
- 建立可重复的本地验收流程。
- 用截图、状态断言、移动端检查证明功能可玩。

必须完成：

- Cocos 经营页结构整理。
- 数据配置接入。
- 资源接入与状态表现。
- 本地 Web 构建与手机尺寸验收。
- Bug 清单与复验记录。

## 阶段计划

### 阶段 0：项目重新定向

目标：建立商业化目标、角色分工和本地任务板。

产出：

- `docs/COMMERCIALIZATION_PLAN.md`
- `docs/LOCAL_TASK_BOARD.md`

验收：

- 能看到当前最重要的工作是经营页，而不是散修功能。
- 能看到产品、美术、开发/测试各自下一步要做什么。
- 后续任务可以在本地管理，不依赖 GitHub issue。

### 阶段 1：经营页重做方案

目标：把经营页从“能玩”提升为“可以作为商店截图和试玩核心”的页面。

产品要完成：

- 经营页从顾客出现到订单完成的完整流程。
- 当前顾客、下一顾客、托盘、微波炉、商品卡之间的规则。
- 哪些信息常驻、哪些只在反馈时出现。

美术/UX 要完成：

- 经营页目标视觉说明。
- 经营页目标图展示。
- 当前经营页差距清单。
- P0 资源清单和布局优先级。

开发/测试要完成：

- 当前经营页代码结构图。
- 哪些改动可以小步实现，哪些需要重构。
- 手机截图验收脚本。

验收：

- 第一次看经营页能知道当前服务对象、订单、操作区域和收入目标。
- 商品、托盘、微波炉、顾客反馈形成清楚闭环。
- UI 不依赖大量文字说明。

### 阶段 2：玩法与数值 V1

目标：让游戏有经营深度，而不是只看图点击。

产品要完成：

- 前 7 天的商品、顾客、订单、设备、升级节奏。
- 每轮目标、优秀目标、失败和部分成交规则。
- 商品售价、成本、补货速度、库存容量。
- 顾客耐心、订单权重、奖励、小费和口碑规则。

美术/UX 要完成：

- 数值反馈的视觉语言：金币、连击、缺货、急躁、完美服务。
- 新机制首次出现时的引导表现。

开发/测试要完成：

- 数据表或配置文件承载数值。
- 至少 7 天可重复测试流程。
- 记录完成单数、收入、流失、净利润的测试结果。

验收：

- 新玩家能完成第一轮。
- 普通玩家会因为调度和库存产生损失。
- 熟练玩家能明显比普通玩家赚更多。
- 首轮利润只支持一个主要升级方向。

### 阶段 3：正式美术资源生产与接入

目标：让画面风格统一，并达到外部试玩的第一眼质量。

产品要完成：

- 只冻结 P0 内容，不继续扩充系统。

美术/UX 要完成：

- 经营页 UI 套件。
- 商品卡正式规范。
- 顾客状态资源。
- 设备、托盘、收银台状态。
- 动效和音效需求。

开发/测试要完成：

- 资源接入表。
- 缺失资源 fallback 策略。
- 手机截图检查文字、按钮、遮挡、点击区域。

验收：

- 经营页截图可以作为项目展示图。
- 主要资源风格一致，没有明显临时素材感。
- 商品缩小到订单尺寸仍能辨认。

### 阶段 4：可上线垂直切片

目标：做出一个可以发给外部玩家试玩的版本。

必须完成：

- 3 至 5 天完整经营循环。
- 新手引导。
- 结算、升级、策略、存档。
- 移动端适配。
- 性能和稳定性检查。
- 已知 bug 清单。

验收：

- 外部玩家不看说明也能开始玩。
- 试玩 10 分钟内能理解成长目标。
- 无明显卡死、遮挡、无法点击、流程死锁。

### 阶段 5：商业化准备

目标：从试玩版本进入发布准备。

暂后处理：

- 广告/IAP 位置。
- 数据埋点。
- 商店素材。
- 隐私和发布清单。
- 多渠道打包。

当前不做这些，避免过早复杂化。

## 当前最高优先级

1. 经营页重做方案。
2. 玩法和数值 V1。
3. P0 美术资源清单。
4. 本地验收标准。

后端、云存档、排行榜、商业化 SDK 暂不进入当前阶段。
