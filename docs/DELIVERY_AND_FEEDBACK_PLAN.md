# 怪兽便利店交付与反馈策划

## 0. 文档目的

本文用于补强当前垂直切片的游玩反馈，重点解决两个问题：

- 玩家完成订单的流程不够像“服务顾客”，更像直接点商品消订单。
- 现有美术资源已经有托盘、加工设备、收银台、顾客表情、订单牌和商品图标，但很多还只是画面装饰，没有承担清晰的玩法职责。

本文不要求大规模重构规则层。首版应优先在 `assets/scripts/presentation/MonsterStorePrototype.ts` 的表现层补出清晰闭环，再逐步把稳定规则下沉到 `core`。

## 1. 当前问题诊断

### 1.1 基于现有结构的判断

当前项目结构显示，垂直切片的主要玩法和表现集中在：

- `assets/scripts/presentation/MonsterStorePrototype.ts`
- `assets/scripts/data/EconomyConfig.ts`
- `assets/scripts/data/ProductCatalog.ts`
- `assets/resources/game-art/**`
- `assets/ui/mockups/**`

`core/GameTypes.ts` 和 `core/OrderGenerator.ts` 仍然比较轻，只定义了简单订单与商品类型。当前更复杂的订单项、加热需求、顾客耐心、库存、微波炉、结算、Toast、付款反馈等，都在 `MonsterStorePrototype.ts` 内部实现。

这说明短期最小方案不应先拆大架构，而应先把表现层的服务流程补完整：订单、拿取、加工、托盘、交付、顾客反馈、收银入账。

### 1.2 玩家是否清楚当前顾客要什么

现状：当前顾客订单气泡已经显示商品图标、加热角标和耐心条，但“当前服务对象”的优先级仍主要靠描边和点击切换反馈。订单信息能读，但不像一个明确的服务任务。

问题：

- 当前顾客与非当前顾客的主次层级还不够强。
- 订单图标缺少数量、缺失、完成、错误状态的统一表达。
- 新玩家可能看得懂图标，却不知道“我要把这些东西送到哪里”。

建议：

- 当前顾客订单气泡放大或高亮，并与托盘状态联动。
- 非当前顾客保留弱化预览，避免干扰。
- 当前订单中的每个商品位显示状态：未完成、已放入托盘、加工中、已完成、错误。

### 1.3 玩家是否清楚商品是否拿对

现状：点击正确商品会 Toast 和弹动，点击错误商品会震动并扣耐心。第一轮前两单有 `showHint` 标记商品。

问题：

- 正确商品目前直接进入已服务状态，缺少“拿起 -> 放托盘 -> 交付”的动作理解。
- 错误反馈主要是文字和震动，没有和订单/托盘强关联。

建议：

- 点商品后先进入 `PICKED` 或直接吸附到托盘，而不是立刻完成订单。
- 正确商品进入托盘格子时，订单气泡对应图标打勾或填充。
- 错误商品被托盘拒绝并弹回，订单气泡或托盘短暂红闪。

### 1.4 玩家是否清楚商品是否需要加工

现状：饭团订单可带 `heated`，订单角标显示“火”，商品卡也能显示“需加热/热食”。点击需要加热的饭团会放入微波炉。

问题：

- 玩家不一定知道“这个饭团不能直接交付，必须先加工”。
- 加工需求和设备入口之间缺少强视觉连线。

建议：

- 需要加工的订单项显示火焰框、未加工锁态、加工中态、完成态。
- 商品卡点击后如果需要加工，商品应飞向微波炉，而不是普通托盘。
- 微波炉在可接收时高亮，错误商品放入时拒绝。

### 1.5 玩家是否清楚加工什么时候开始、什么时候完成

现状：微波炉有 `idle/heating/ready`，加热时显示进度条，完成时有 Toast、发光和完成标记。

问题：

- 微波炉美术已经存在 `idle/heating/ready` 单独资源和 unified 资源，但当前更多靠程序叠加效果，状态资源没有完全承担玩法信息。
- 完成提示仍可能依赖文字，玩家视线不一定在设备上。

建议：

- `PROCESSING` 时设备窗口发光、进度环/条显示。
- 快完成时设备轻抖、冒气或灯闪。
- `READY` 时成品弹出或放在设备口，设备处于阻塞态，直到玩家取走。

### 1.6 玩家是否清楚托盘的作用

现状：托盘/交付台已作为画面设备显示，有完成勾，但没有规则意义。商品点击后直接服务顾客，托盘不是判定节点。

问题：

- 托盘是最有潜力成为核心交付节点的资源，但当前像装饰。
- 玩家没有“组装订单”的操作和成就感。
- 服务闭环少了关键一步：把东西放到托盘后再交付。

建议：

- 托盘成为当前订单的唯一交付判定区。
- 商品先进入托盘格，托盘判断是否满足当前顾客订单。
- 订单完成后托盘发光，玩家点击托盘或交付按钮完成交付。

### 1.7 玩家是否清楚什么时候可以交付

现状：当前版本主要是商品全部点击正确后自动付款，不需要玩家交付。

问题：

- 自动付款省操作，但削弱“我完成了一单”的爽感。
- 托盘和收银台被弱化，服务闭环不完整。

建议：

- 商品全部放入托盘且匹配当前订单时，托盘进入 `READY`。
- `READY` 时托盘发光、出现箭头或 Deliver 标识。
- 玩家点击托盘完成交付。为了不增加低龄复杂度，首版只要求点击托盘，不做扫码/收银二次交互。

### 1.8 玩家是否清楚成功、错误、超时发生了什么

现状：

- 成功：Toast、顾客开心、付款数字、金币反馈。
- 错误：商品卡震动，顾客耐心下降。
- 超时：顾客生气离开，连击中断。

问题：

- 成功缺少“商品飞向顾客、订单完成盖章、顾客强表情”的完整组合。
- 错误没有形成“错在哪里”的定位反馈。
- 超时反馈应让玩家看见耐心从危险到失败的过渡。

建议：

- 成功：托盘闪、商品飞、气泡盖章、顾客开心、金币飞、Combo。
- 错误：托盘红闪、错误商品弹回、订单对应缺失位晃动。
- 超时：耐心条红闪、顾客表情切 `urgent -> angry -> leave`，订单气泡裂纹或灰掉。

### 1.9 经营便利店闭环是否成立

当前有库存、仓库、货架、微波炉、收银台、经营中心、结算和升级，但营业中的手感仍偏“点图标结算”。便利店闭环应该是：

顾客提出需求 -> 玩家从货架取货 -> 需要时加工 -> 托盘组装 -> 交付 -> 顾客反应 -> 收银入账 -> 下一位推进。

当前最缺的是“托盘组装与交付”。这是让美术资源从摆设变成玩法的关键。

### 1.10 当前像摆设的资源与应承担功能

| 资源 | 当前问题 | 应承担的玩法功能 |
|---|---|---|
| 托盘 / delivery-pad | 主要是装饰，有勾但没有规则含义 | 当前订单组装、匹配判定、交付入口 |
| 货架 / 背景货架 | 背景表现强，但商品取货仍在底部卡片 | 可作为商品来源的视觉语义；卡片点击时商品从卡片/货架飞出 |
| 微波炉 | 已有状态反馈，但还不是强入口 | 加工需求的唯一处理节点，显示可接收、加工、完成、阻塞 |
| 收银台 | 有金额显示，但偏结果展示 | 付款入账反馈、金币飞向 HUD、小费/Combo 表现 |
| 订单气泡 | 已显示商品，但状态层级不足 | 当前任务面板，显示缺失、加工、完成、错误 |
| 怪兽表情 | 有 waiting/happy/urgent/angry 资源 | 服务状态机核心反馈，不只是站桩 |
| 商品图标 | 已用于卡片和订单 | 取货、托盘、飞行动画、加工前后状态都应复用 |
| 背景装饰 | 目前只提供氛围 | 不参与点击，避免干扰；可承接日后升级/装修 |

## 2. 目标体验

核心体验定义为：

```text
顾客进店 -> 出现订单 -> 玩家取货/加工 -> 放入托盘组装订单 -> 交付给顾客 -> 顾客反馈 -> 获得分数/小费 -> 下一位顾客推进
```

这个流程必须让玩家一眼看懂，并且每一步都有明确反馈。

### 2.1 90 秒营业反馈节奏

#### 开局 3 秒

目标：玩家知道今天开始营业、目标是什么、哪里是操作区。

反馈：

- 顶部 HUD 出现并轻弹：营业额、时间、金币。
- 第一个当前顾客气泡亮起，托盘显示 2 个淡占位格。
- 轻提示只出现一次：“按订单取货，放到托盘交付”。
- 不弹长说明，不遮挡订单。

#### 顾客出现

目标：玩家知道谁是当前服务对象。

反馈：

- 当前顾客从下方或门口位置轻弹入场。
- 当前顾客脚下或气泡边缘发光。
- 非当前顾客订单气泡弱化 60% 透明度或缩小。
- 急性顾客使用黄色/橙色表情、耐心条较短。

#### 订单出现

目标：玩家知道要拿哪些商品。

反馈：

- 订单图标按顺序弹入气泡。
- 每个图标有数量和状态：`0/1`、`1/2`、`加工中`。
- 需要加热的商品带火焰框，并与微波炉短暂连线或微波炉高亮。

#### 商品操作

目标：玩家知道自己拿起了什么、能放哪里、放错会怎样。

反馈：

- 点击正确商品：商品从卡片飞到托盘格，卡片弹一下。
- 点击错误商品：卡片左右震，托盘不接收，当前订单缺失位红闪。
- 点击售罄商品：商品卡灰色，顾客气泡中对应商品出现缺货警告。
- 点击非当前订单但下一订单需要的商品：首版不提前入托盘，避免复杂；P1 再做备餐位。

#### 加工流程

目标：玩家知道商品正在加工、即将完成、已完成。

反馈：

- 点击需要加热商品：商品飞入微波炉，设备进入 `PROCESSING`。
- 加工中：进度条/环围绕设备显示，订单图标显示“加工中”。
- 剩余 20%：微波炉轻微抖动、冒气、灯闪。
- 完成：成品从设备口弹出，设备进入 `READY/BLOCKED`，订单图标显示可取。

#### 托盘组装

目标：玩家知道托盘里还缺什么、哪些已经正确。

反馈：

- 空托盘显示淡占位格。
- 正确商品吸附到对应格子。
- 缺失格子显示虚线图标或 `+`。
- 完成后托盘发光、弹跳，出现交付箭头。

#### 交付瞬间

目标：玩家获得明确爽感。

反馈：

- 点击 READY 托盘。
- 托盘闪光。
- 商品按顺序飞向当前顾客。
- 顾客开心跳动。
- 订单气泡盖章 `DONE`。
- 金币/小费从顾客或收银台飞向 HUD。

#### 顾客离开

目标：玩家知道赚了多少分/小费。

反馈：

- 顾客头顶弹出 `+收入`、`小费 +N`、`Combo xN`。
- 开心顾客离开或淡出，下一位顾客顺滑补位。

#### 营业结束

目标：玩家知道本轮表现好坏。

反馈：

- 结算面板显示：完成订单、流失、营业额、成本、净利润、评分。
- 本轮失误类型可简短列出：错单、超时、缺货。
- 明确入口：进货准备 / 升级 / 下一轮策略。

## 3. 托盘玩法重新定义

托盘必须成为核心交付节点，而不是装饰。

P0 采用“单托盘锁定顾客”规则，而不是“切换顾客时托盘内容瞬间变化”。原因是后者虽然数据上可行，但视觉上会像商品突然消失或变成另一份订单，玩家容易误以为 bug。

首版规则：

- 画面上只有一个托盘。
- 托盘为空时，玩家可以选择任意顾客作为服务对象。
- 第一件商品进入托盘，或商品进入微波炉后，系统建立 `serviceLockCustomerId`。
- 锁定期间，托盘和微波炉只服务这个顾客。
- 玩家仍然可以点击切换顾客查看订单，但不能把其他顾客的商品放入当前锁定托盘。
- 交付成功、顾客离开、订单失败或手动清空后解除锁定。
- 微波炉与托盘共享同一个服务锁，避免热饭团完成后无法解释应该进入哪份订单。

### 3.1 托盘状态

| 状态 | 含义 | 可操作 |
|---|---|---|
| `EMPTY` | 空托盘，没有商品 | 可接收当前订单商品 |
| `PARTIAL` | 已放入部分正确商品 | 可继续接收缺失商品 |
| `READY` | 当前订单已满足 | 点击托盘交付 |
| `WRONG` | 放入错误商品或错误加工态 | 拒绝或短暂错误反馈 |
| `FULL` | 托盘格满但未满足订单 | 不可交付，提示缺少正确商品 |
| `DELIVERING` | 正在交付动画中 | 暂停输入 |

### 3.2 托盘视觉反馈

| 场景 | 视觉反馈 |
|---|---|
| 空托盘 | 显示淡淡占位格，格子数量等于当前订单所需件数 |
| 放入正确商品 | 商品吸附到托盘格子，订单气泡对应图标打勾 |
| 放入错误商品 | 托盘抖动，商品弹回，托盘边框红闪 |
| 订单未完成 | 托盘底部短提示“还缺 1 件”，1 秒后消失 |
| 订单完成 | 托盘发光、弹跳，出现交付箭头或 `DELIVER` |
| 交付成功 | 托盘商品飞向顾客，托盘清空 |
| 交付失败 | 顾客疑惑/生气，托盘轻震，不清空或退回错误商品 |

### 3.3 判定逻辑

建议新增表现层状态：

```ts
type TrayState = 'EMPTY' | 'PARTIAL' | 'READY' | 'WRONG' | 'FULL' | 'DELIVERING';

type TrayItem = {
    productId: ProductId;
    heated?: boolean;
    source: 'shelf' | 'microwave';
};
```

匹配规则：

- 托盘只匹配锁定顾客订单。若托盘为空且无加工中商品，则首次正确操作会锁定当前选中顾客。
- 切换顾客不会切换托盘内容；托盘继续显示锁定顾客的商品，并用边框颜色或短标签提示“正在服务左侧顾客/右侧顾客”。
- 锁定期间如果玩家尝试给其他顾客放商品，托盘拒绝并提示“先交付当前托盘”。
- 允许重复商品，但必须订单中也存在重复需求；当前首版订单多为不重复，但结构应支持。
- 未加工商品不能满足 `heated: true` 的订单项。
- 加工完成商品可以满足普通 `rice-ball` 吗：首版建议不允许，避免“热饭团算普通饭团”的歧义；后续可作为宽容策略。
- 半成品不允许放入托盘；加工中商品在微波炉内占位。
- 错误商品首版直接弹回并不消耗库存；如果已经消耗库存，需要立刻归还，避免挫败过强。
- 托盘容量等于当前订单项数量，最小 1，最大随切片 2 至 3。

### 3.4 交付逻辑

首版建议：

```text
空托盘 + 无加工 -> 当前选中顾客可开始服务
点击商品 -> 若匹配当前订单且不需加工 -> 加入托盘
点击需加工商品 -> 进入微波炉
点击 ready 微波炉 -> 加入托盘
托盘匹配完整 -> READY
点击 READY 托盘 -> 交付成功
交付/失败/离开 -> 清空托盘并解除服务锁
```

不要先做拖拽，点击即可验证闭环。拖拽可以作为 P1 手感升级。

## 4. 顾客反馈系统

顾客是项目特色，不能只是站在画面里。建议将当前 `CustomerMood` 扩展为更明确的服务状态机，视觉资源先复用现有四种表情。

### 4.1 状态机

| 状态 | 触发条件 | 视觉表现 | 动画表现 | 音效/震动建议 | UI 提示 | 分数/时间影响 |
|---|---|---|---|---|---|---|
| `ENTER` | 新顾客生成 | 顾客从下方/门口出现，气泡未完全展开 | 入场弹跳 0.25s | 门铃/轻快音 | 无或短提示“新顾客” | 不扣耐心或延迟 0.5s |
| `ORDERING` | 入场完成 | 订单气泡展开，商品图标依次弹出 | 气泡缩放，图标 stagger | 纸张/提示音 | 当前顾客描边发亮 | 耐心开始倒计时 |
| `WAITING` | 正常等待 | waiting 表情，耐心条绿色 | 呼吸、眨眼、左右看 | 轻环境音 | 无 | 正常扣耐心 |
| `IMPATIENT` | 耐心 < 35% | urgent 表情，汗滴/抖动 | 身体小抖，耐心条黄/橙 | 心跳或急促提示 | “快生气了”短提示 | 完美小费不可得 |
| `ANGRY` | 耐心接近 0 或错误多次 | angry 表情，冒火 | 快速抖动、红闪 | 低沉音/短震 | 订单气泡红闪 | 即将流失，连击危险 |
| `HAPPY` | 收到正确订单 | happy 表情，眼睛弯/月牙 | 跳动、手臂上扬 | 叮咚/金币音 | `DONE`、收入数字 | 加分、小费、Combo |
| `CONFUSED` | 错误交付 | 疑惑/皱眉，当前可用 angry 或 urgent 替代 | 后仰/摇头 | 错误音/轻震 | “订单不对” | 扣耐心，Combo 断 |
| `LEAVE` | 完成、失败、部分成交后 | 顾客淡出或下移离开 | 移出屏幕 0.3s | 门铃低音/离开音 | 根据结果显示收入/流失 | 生成下一顾客或结算 |

### 4.2 资源使用建议

当前已有：

- `normal/waiting,happy,urgent,angry`
- `impatient/waiting,happy,urgent,angry`
- `bulk-shopper/waiting,happy,urgent,angry`

首版不需要新增逐帧动画。使用现有 PNG 加程序动效：

- 等待：`scaleY` 轻微呼吸，周期 1.8s。
- 眨眼：如果没有拆眼资源，先用 0.08s 纵向压缩模拟。
- 着急：水平小抖 + 汗滴程序圆点。
- 生气：红色半透明覆盖 + 蒸汽粒子。
- 开心：整体跳动 + 星星/金币粒子。

## 5. 订单反馈系统

### 5.1 当前订单气泡

当前顾客订单气泡应是最高优先级信息面板。

每个订单项建议结构：

```text
商品图标
数量：x1 / 1/2
加工状态：需加热 / 加工中 / 已完成
完成状态：缺失 / 已入托盘 / 已交付
错误提示：短暂红框，不常驻
```

示例：

```text
怪兽薯片 x1        未完成
热饭团 x1          加工中
星星糖果 x2        已完成 1/2
```

图标状态：

- 未完成：正常图标，浅描边。
- 已入托盘：绿色勾，图标轻亮。
- 加工中：火焰进度小环。
- 加工完成待取：黄色闪光。
- 错误：红色抖动 0.25s。

### 5.2 下一订单预览

下一订单预览用于规划，不应抢当前操作视线。

建议：

- 非当前顾客气泡缩小 90% 或透明度 65%。
- 只显示商品图标和加工标记，不显示详细数量状态。
- 当前顾客切换后，预览气泡变为当前气泡并展开。
- 不在首轮前 10 秒强调下一单，避免新手负担。

### 5.3 主次层级

| 信息 | 层级 | 表现 |
|---|---|---|
| 当前订单 | 最高 | 大气泡、强描边、状态完整 |
| 当前托盘 | 最高 | 高亮格子、交付箭头 |
| 当前设备 | 高 | 加工时强反馈 |
| 下一订单 | 中 | 弱化预览 |
| 库存/价格 | 中低 | 商品卡角标，不抢订单 |
| 经营信息 | 低 | HUD 和结算，不干扰操作 |

## 6. 商品与加工设备反馈

### 6.1 商品状态

| 状态 | 含义 | 反馈 |
|---|---|---|
| `ON_SHELF` | 在货架/商品卡可取 | 卡片正常、库存角标 |
| `PICKED` | 被玩家点击拿起 | 商品飞出，卡片弹动 |
| `ON_TRAY` | 已放入托盘 | 托盘格显示商品，订单项打勾 |
| `PROCESSING` | 加工中 | 商品隐藏在设备中，订单项显示进度 |
| `PROCESSED` | 加工完成 | 设备口显示成品或 ready 徽章 |
| `DELIVERED` | 已交付 | 商品飞向顾客，订单盖章 |
| `RETURNED` | 放错退回 | 商品弹回卡片或消失不消耗 |

### 6.2 从货架到托盘的反馈流

```text
点击商品卡
-> 卡片弹动
-> 商品复制体从卡片飞向托盘/设备
-> 托盘/设备判定
-> 成功吸附或错误弹回
-> 订单气泡对应项更新
```

### 6.3 加工设备状态

| 状态 | 含义 | 反馈 |
|---|---|---|
| `IDLE` | 空闲 | 普通状态，低亮度 |
| `ACCEPTING` | 当前订单有可加工商品 | 设备边缘高亮，火焰按钮亮 |
| `PROCESSING` | 加工中 | 进度条/环、窗口发光 |
| `READY` | 完成待取 | 成品弹出，设备闪光 |
| `BLOCKED` | 成品未取，不能继续加工 | 设备红/黄提示，点击提示“先取出” |

### 6.4 加工设备必须反馈

- 放入商品时：设备亮起，商品飞入窗口。
- 加工时：显示进度环或进度条。
- 快完成时：轻微抖动、冒气、提示音。
- 完成时：成品弹出，声音明确。
- 未取走时：设备显示堵塞或等待，不接收新商品。
- 放错商品时：设备拒绝，抖动，错误商品弹回。

## 7. 交付爽感设计

交付成功必须是一套组合反馈，而不是单个 Toast。

### 7.1 普通成功

触发：托盘满足订单，但耐心低于完美阈值。

反馈：

- 托盘白光闪一下。
- 商品飞向顾客。
- 顾客切 happy 并轻跳。
- 订单气泡盖章 `DONE`。
- 收银台显示本单金额。
- 金币飞向顶部营业额。

### 7.2 快速成功

触发：耐心高于 60%，没有错误。

反馈：

- 托盘金色闪光。
- 顾客更大幅度跳动。
- 弹出“小费 +N”。
- Combo 增加。
- 收银台金额跳动更明显。

### 7.3 完美成功

触发：快速成功 + 没有等待过危险区 + 没有错误商品/加工错误。

反馈：

- `PERFECT` 印章。
- 星星粒子。
- 小费金币数量更多。
- 顾客开心动作更夸张。
- Combo 条短暂发光。

### 7.4 错误交付

触发：托盘内容与当前订单不匹配时点击交付。

反馈：

- 托盘红闪并震动。
- 错误商品格子标红。
- 顾客进入 `CONFUSED` 或 `ANGRY`。
- 耐心扣除固定值，Combo 断。
- 不清空正确商品，错误商品退回。

### 7.5 超时失败

触发：顾客耐心归零。

反馈：

- 顾客切 angry。
- 订单气泡灰掉/破裂。
- 托盘中属于该顾客的商品退回或清空。
- 顾客离开，显示 `流失 -1`。
- Combo 断，评分降低。

## 9. P0 / P1 / P2 实现优先级

### P0：必须立刻做，否则垂直切片不好玩

- 当前订单气泡清晰显示：商品、数量、加工、完成/缺失状态。
- 商品点击后不直接完成，先进入托盘或设备。
- 托盘可放置并参与当前订单判定。
- 托盘满足订单后可点击交付。
- 成功、错误、超时三类反馈必须明显不同。
- 顾客至少有等待、开心、生气三种状态和程序动作。
- 微波炉至少有放入、加工进度、完成待取、阻塞反馈。
- 90 秒或当前营业时长结束后，有清楚结算。

### P1：增强爽感

- Combo 连击与完美服务评价。
- 小费金币飞行动画。
- 下一订单预览弱化显示，让熟练玩家提前规划。
- 顾客个性动作：急客抖动、大单客慢吞吞等。
- 完美交付 `PERFECT` 盖章。
- 连续服务节奏加速，下一位顾客顺滑切入。
- 拖拽操作或点击后短暂“拿起”态。

### P2：商业化和长期体验

- 更多顾客类型：挑剔、健忘、幸运等。
- 更多商品加工链：冰镇、组合餐、包装。
- 道具：自动托盘、快速加热、安抚顾客。
- 升级系统扩展：多托盘位、双微波炉、收银效率。
- 装修系统：店铺外观、货架皮肤、设备皮肤。
- 广告设计：复活流失顾客、临时补货、延长营业、双倍小费。

## 10. 最小可落地方案

目标：不大改 `core`，优先在 presentation 层补出一套可玩的交付闭环。

### 10.1 新增/修改脚本文件

| 文件 | 操作 | 职责 |
|---|---|---|
| `assets/scripts/presentation/MonsterStorePrototype.ts` | 修改 | 增加托盘状态、托盘绘制、商品进入托盘、交付判定、反馈动画 |
| `assets/scripts/core/GameTypes.ts` | 可选小改 | 后续沉淀 `OrderItem`、`TrayItem` 类型；P0 可不动 |
| `assets/scripts/data/ProductCatalog.ts` | 可选小改 | 增加是否可加工、加工后状态标签；P0 可在 presentation 写映射 |
| `docs/DELIVERY_FEEDBACK_TODO.md` | 新增 | 拆成执行任务 |

首版不要新增多个组件脚本，避免 Cocos 节点绑定成本。等闭环验证成立，再拆 `TrayPresenter`、`CustomerPresenter`、`FeedbackFx`。

### 10.2 数据结构建议

在 `MonsterStorePrototype.ts` 内先新增：

```ts
type TrayState = 'EMPTY' | 'PARTIAL' | 'READY' | 'WRONG' | 'FULL' | 'DELIVERING';

type TrayItem = {
    productId: ProductId;
    heated?: boolean;
};

type ServiceLock = {
    customerId: number;
    sideLabel: string;
};

type ProductFxState = {
    productId: ProductId;
    fromX: number;
    fromY: number;
    toX: number;
    toY: number;
    remaining: number;
};
```

已有 `OrderItem` 可以继续使用：

```ts
type OrderItem = {
    productId: ProductId;
    heated?: boolean;
};
```

### 10.3 事件流建议

#### 普通商品

```text
handleProductTap(productId)
-> 若托盘/微波炉已锁定其他顾客，拒绝并提示
-> 若未锁定，锁定当前选中顾客
-> 判断商品是否在当前订单缺失项中
-> 判断是否需要加工
-> consumeStock(productId)
-> addTrayItem({ productId })
-> updateOrderItemState
-> if tray complete: trayState = READY
```

#### 加工商品

```text
handleProductTap(rice-ball heated)
-> 若未锁定，锁定当前选中顾客
-> 若已锁定其他顾客，拒绝
-> consumeStock
-> microwave = PROCESSING
-> 订单项显示加工中
-> microwave READY
-> handleMicrowaveTap
-> 若托盘仍属于同一顾客，将热饭团加入托盘
-> addTrayItem({ productId: 'rice-ball', heated: true })
-> if tray complete: trayState = READY
```

#### 交付

```text
handleTrayTap()
-> if READY: playDeliverFx
-> finishCustomerOrder
-> clearTray
-> clearServiceLock
-> replaceCustomer
-> if not ready: show missing hint
```

### 10.4 UI 节点绑定建议

当前项目大多通过程序创建节点，首版继续使用程序节点：

- `WorkstationBand/TrayArea`
- `TraySlot-0..N`
- `TrayItem-{productId}`
- `OrderItem-{index}`
- `Microwave`
- `PaymentBurst`

不必先在 Cocos 场景里手动绑定节点，减少编辑器误差。

### 10.5 可以假数据/占位的地方

- 下一订单预览可以先复用非当前顾客订单，不必新增队列系统。
- `CONFUSED` 表情可以复用 `urgent` 或 `angry`。
- 音效可以先只写触发点，不接真实音频。
- 商品飞行动画可以先用复制图标节点，不做真实曲线。
- Combo UI 可以先用数字弹出，不做完整连击条。

### 10.6 必须真判定的地方

- 托盘内容是否匹配当前顾客订单。
- 托盘和微波炉的服务锁不能混乱：锁定 A 顾客时不能把 B 顾客商品放入托盘。
- 加工前商品不能满足加工订单。
- 托盘 READY 后才能交付成功。
- 错误商品不能静默成功。
- 超时必须清理该顾客相关托盘/设备状态。
- 库存消耗和退回规则必须明确。

## 11. 验收标准

试玩时玩家应该能做到：

- 3 秒内看懂当前顾客要什么。
- 5 秒内知道商品可以放到托盘或微波炉。
- 第一次成功交付时有明显正反馈。
- 放错商品时立刻知道错在哪里。
- 等待太久时知道顾客正在变生气。
- 托盘不再像摆设，而是订单完成的核心区域。
- 加工商品从放入、加工中、完成待取都能看懂。
- 90 秒结束时知道自己表现好坏。
- 玩家能说出完整服务闭环：取货、加工、放托盘、交付、收钱。

## 12. Blocking Issues

当前没有阻止 P0 落地的硬阻塞，但有几个结构风险：

1. `MonsterStorePrototype.ts` 已经承担玩法、表现、资源加载、经济系统、UI 绘制等全部职责，继续加托盘会更臃肿。P0 可以接受，P1 后建议拆表现辅助类。
2. `core/GameTypes.ts` 的 `Order` 仍是旧的 `productIds` 模型，而实际运行使用 `OrderItem { productId, heated }`。短期可继续在 presentation 内实现，后续应统一。
3. 托盘资源存在，但没有多状态图。P0 不需要新图，可用程序格子、描边、发光完成。
4. 目前没有音频系统接入。文档中的音效先作为触发点，不应阻塞视觉反馈。
5. 当前输入以点击为主，不是拖拽。P0 应避免突然改为拖拽，先用点击商品自动入托盘验证闭环。

## 13. 结论

当前垂直切片最应该补的不是更多商品、更多顾客或更多经营系统，而是把托盘变成订单交付核心。只要托盘承担“组装、判定、交付”三件事，现有美术资源会自然进入玩法闭环：

```text
订单气泡告诉你要什么
商品卡/货架提供商品
微波炉处理加工要求
托盘承接组装和交付
怪兽表达结果
收银台表达收入
```

这会直接影响玩家是否觉得游戏“好玩”，因为它把当前的看图点击变成了便利店服务流程。
