## 当前确认

经营页 V1 采用“订单气泡 READY 交付”流程。

```text
顾客提出订单
玩家判断当前服务对象
玩家从商品区取货
普通商品飞入当前顾客订单气泡
需要加工的商品进入微波炉
加工完成后商品飞入当前顾客订单气泡
订单气泡全部备齐后进入 READY
点击当前顾客或 READY 订单气泡完成交付
顾客反馈
收银入账
下一位顾客推进
```

保留的玩法原则：

- 商品不能点击后直接收钱。
- 多商品订单必须有“还缺什么 / 已备齐”的过程。
- 切换顾客时不能混单。
- 热食仍需经过微波炉。
- 收银台仍负责入账反馈。

删除的表现目标：

- 独立交付托盘。
- 传送带式备餐队列。
- 独立 `DELIVERED` 交付按钮。

## 目的

这份文档只定义“经营页”怎么玩。它不是全游戏数值表，也不是美术规范。它的作用是让美术、开发和测试都知道经营页必须服务哪些玩法信息。

经营页的商业化目标：

- 玩家第一次看到画面，能立刻理解这是怪兽便利店在接待顾客。
- 玩家 1 分钟内能完成第一单。
- 普通玩家能完成一轮，但会因为服务顺序、加工、缺货或错误产生损失。
- 熟练玩家能通过预判、并行加工和库存规划赚更多钱。
- 经营页截图能展示游戏卖点：怪兽顾客、订单气泡、商品准备、时间压力、收入反馈。

## 核心体验

经营页的核心体验是：

```text
看订单
选顾客
点商品
补齐订单气泡
等待或取出热食
订单 READY
点击交付
收钱
推进下一位顾客
```

这个流程必须通过画面、动效和数值反馈表达出来，不能只靠 Toast 文字解释。

## 第一眼必须看懂什么

玩家进入经营页 3 秒内必须能看懂：

1. 现在正在营业，并且有倒计时。
2. 当前有两位顾客在等待。
3. 哪位顾客是当前服务对象。
4. 当前服务对象要什么商品。
5. 底部商品卡可以点击。
6. 订单气泡是备齐商品和 READY 交付的位置。
7. 微波炉用于处理带火焰需求的商品。
8. 收银台/顶部收入会在成功交付后变化。

如果玩家第一眼需要读大段文字，经营页就没有达标。

## 页面区域职责

### 1. 顶部 HUD

玩法职责：

- 显示本轮时间压力。
- 显示本轮经营结果的核心数值。
- 显示暂停入口。

P0 常驻信息：

- 剩余时间。
- 当前营业额或金币。
- 满意度/口碑/连击中的一个核心反馈位。
- 暂停按钮。

设计规则：

- 时间是最高优先级，必须比其他数值更醒目。
- 收入变化要与收银台付款动效联动。
- 不要堆太多经营中心信息，经营页只显示当下决策需要的信息。

### 2. 顾客区

玩法职责：

- 显示等待顾客。
- 显示订单。
- 显示耐心。
- 显示当前服务对象。
- 显示成功、急躁、生气、离开等反馈。

P0 规则：

- 同屏最多 2 位顾客。
- 玩家可以点击顾客切换服务对象。
- 当前顾客必须有明显高亮。
- 非当前顾客仍然显示订单，作为预判信息。
- 每位顾客有独立耐心条。
- 当前顾客的订单气泡是主要操作反馈区。

顾客状态：

| 状态 | 触发 | 玩法意义 | 必须反馈 |
| --- | --- | --- | --- |
| waiting | 正常等待 | 可服务 | 默认表情、订单气泡 |
| urgent | 耐心低于警戒线 | 需要优先处理 | 急躁表情、耐心条变色 |
| happy | 订单完成 | 正反馈 | 开心表情、付款反馈 |
| angry | 错误或超时 | 惩罚反馈 | 生气表情、连击中断 |
| leaving | 完成/失败后的退场 | 队列推进 | 订单气泡弱化或盖章 |

### 3. 订单气泡

玩法职责：

- 告诉玩家顾客需要什么。
- 显示每个商品的准备状态。
- 显示是否需要加工。
- 显示该订单是否已经 READY 或完成。
- 作为 READY 后的交付入口。

订单项状态：

| 状态 | 含义 | UI/动效要求 |
| --- | --- | --- |
| missing | 尚未准备 | 商品图标正常显示，槽位未点亮 |
| filled | 已放入订单气泡 | 商品图标亮起、打勾或弹跳 |
| processing | 正在微波炉加工 | 商品图标显示火焰/进度状态 |
| ready_from_device | 设备加工完成，等待填入订单 | 微波炉 READY，同时订单图标提示可取 |
| delivered | 已交付 | 订单盖章或商品位完成 |
| error | 玩家点错或交付失败 | 对应商品位红闪 |

订单整体状态：

| 状态 | 含义 | 玩家动作 | 必须反馈 |
| --- | --- | --- | --- |
| empty | 尚未准备任何商品 | 点击该顾客需要的商品 | 槽位清楚显示缺什么 |
| partial | 已准备部分商品 | 继续补齐缺失商品 | 已填项与缺失项明显区分 |
| ready | 订单已满足 | 点击订单气泡或当前顾客交付 | 发光、READY 标识、轻微呼吸动效 |
| blocked | 尝试混单或错误交付 | 回到正确服务对象或补齐缺失 | 红闪、震动、局部提示 |
| done | 已完成 | 无 | DONE 盖章、顾客开心、收银反馈 |

P0 订单数量：

- 第 1 天主要 1 件商品订单。
- 逐步出现 2 件商品订单。
- 热食订单首次出现时必须明确标火焰。

后续可扩展：

- 3 至 4 件订单。
- 数量重复。
- 冰镇等第二加工需求。

### 4. 商品区

玩法职责：

- 提供玩家可取商品。
- 显示库存。
- 显示是否未解锁、售罄、需要加工、当前订单需要。

点击商品的判定顺序：

1. 商品未解锁：不可点击或提示营业后解锁。
2. 商品货架库存为 0，但仓库还有：触发补货等待。
3. 商品货架和仓库都为 0：触发缺货逻辑。
4. 当前服务顾客不需要该商品：判定错误。
5. 当前服务顾客需要该商品且不需加工：商品进入当前顾客订单气泡。
6. 当前服务顾客需要该商品且需加工：商品进入微波炉。

商品卡必须表达：

- 商品本身。
- 当前货架库存。
- 是否当前订单需要。
- 是否需要加热。
- 是否锁定/售罄。

商品卡不应承担：

- 复杂经营解释。
- 大段文字规则说明。
- 未来多天预测。

### 5. 微波炉

玩法职责：

- 处理带加热需求的商品。
- 制造“等待期间处理其他订单”的调度空间。
- 让高利润商品带来操作压力。

P0 规则：

- 只有一个处理位。
- 只有需要加热的商品能进入。
- 加热中不能接收第二件商品。
- 加热完成后必须点击微波炉，把成品填入对应订单气泡。
- 微波炉加工会记录所属顾客，不能把成品填给错误顾客。

状态：

| 状态 | 触发 | 玩家动作 | 必须反馈 |
| --- | --- | --- | --- |
| idle | 空闲 | 可放入热食 | 设备可用 |
| heating | 商品加工中 | 等待或处理其他顾客 | 进度条/发光 |
| ready | 加工完成 | 点击取出并填入订单气泡 | READY 光效、提示 |
| locked | 正在处理另一位顾客的热食 | 不能放入其他热食 | 拒绝/震动 |

### 6. 服务锁与混单规则

服务锁不是一个额外 UI 物件，而是一条玩法规则：已经开始准备某位顾客的订单后，该批商品必须继续归属于这位顾客，直到该订单完成、失败或清理。

P0 规则：

- 订单气泡为空时，第一件正确商品会锁定当前顾客的该笔订单。
- 微波炉开始加工时，也会锁定该热食所属顾客。
- 锁定后切换顾客不会改变已准备商品的归属。
- 锁定期间，不能把其他顾客的商品填入这笔订单。
- 当前顾客的订单气泡满足需求后进入 READY。
- 只有 READY 订单气泡或当前顾客区域可触发交付。

混单反馈：

- 商品不消耗。
- 订单气泡不改变。
- 错误目标局部红闪。
- 当前应完成的订单气泡轻微提示。
- 顾客耐心可小幅减少。

P0 不做：

- 独立交付托盘。
- 多备餐位。
- 提前为下一位顾客备餐。
- 拖拽摆放。
- 订单气泡随顾客切换瞬间变成另一笔已准备内容。

### 7. 收银台

玩法职责：

- 显示付款结果。
- 强化“完成订单赚到钱”的反馈。
- 将订单交付和顶部收入连接起来。

P0 规则：

- 顾客完成订单后自动付款。
- 不增加扫码或二次结账操作。
- 收银台只承担结果反馈，不增加新操作。

必须反馈：

- 收银台闪光或弹动。
- 金币/收入数字出现。
- 顶部营业额增加。

## 一单完整流程

### 普通商品订单

```text
顾客出现并显示订单
玩家确认当前顾客
玩家点击订单需要的普通商品
商品从商品卡飞向当前顾客订单气泡
订单商品位亮起
订单气泡进入 READY
玩家点击 READY 订单气泡或当前顾客
商品交付给顾客
顾客开心
收银台入账
顶部营业额变化
该顾客离开并生成下一位
```

### 多商品订单

```text
顾客订单显示 2 件商品
玩家点击第一件
第一件进入订单气泡，订单状态 partial
玩家点击第二件
第二件进入订单气泡，订单状态 READY
玩家点击 READY 订单气泡或当前顾客交付
```

多商品订单的关键压力：

- 玩家必须确认还缺什么。
- 如果货架库存不足，会触发补货或缺货判断。
- 如果另一个顾客更急，玩家要决定是否先切换服务对象。

### 热食订单

```text
顾客订单显示带火焰商品
玩家点击对应商品
商品飞向微波炉
微波炉进入 heating
订单商品位显示 processing
玩家等待时可观察或处理其他顾客
微波炉 ready
玩家点击微波炉
热食飞入所属顾客订单气泡
订单气泡 READY
玩家点击 READY 订单气泡或当前顾客交付
```

热食订单的关键压力：

- 微波炉占用制造排队。
- 玩家可以利用等待时间查看或处理其他订单。
- 忘记取出会阻塞后续热食。
- 微波炉成品必须回到所属订单，不能混给其他顾客。

## 错误与失败规则

### 点错商品

触发：

- 当前服务顾客不需要该商品。
- 当前服务顾客需要该商品，但商品状态不匹配，例如未加热热食无法直接填入订单。

结果：

- 不进入订单气泡。
- 商品不消耗。
- 顾客耐心减少。
- 连击中断。
- 商品卡震动。
- 对应订单气泡或商品位红闪。
- 顾客短暂生气或急躁。

设计目的：

- 让玩家知道自己错在哪里。
- 惩罚准确性，而不是只扣分无反馈。

### 混单

触发：

- A 顾客订单已经开始准备，玩家切到 B 顾客并尝试把 B 顾客商品填入 A 订单。
- 微波炉正在处理 A 顾客热食，玩家尝试用成品完成 B 顾客订单。

结果：

- 商品不消耗。
- 订单气泡不改变。
- 提示先完成当前准备中的订单。
- 相关订单气泡红闪或震动。

设计目的：

- 保证订单归属清楚。
- 避免商品突然消失或归属不明。

### 缺货

触发：

- 需要的商品货架为 0。
- 仓库有库存时进入补货等待。
- 仓库也为 0 时判定售罄。

结果：

- 仓库有货：提示补货中，暂时不能拿。
- 完全售罄：若订单无法完成，顾客不满或离开。
- 多商品订单已部分准备时，顾客可为已拿商品部分付款，但不给小费，连击中断。

设计目的：

- 让营业前采购和库存结构影响营业。
- 允许玩家因为经营决策而损失顾客。

### 超时

触发：

- 顾客耐心降到 0。

结果：

- 顾客生气离开。
- 连击中断。
- 评价降低。
- 若订单气泡或微波炉正在服务该顾客，相关状态清理。

设计目的：

- 制造服务顺序压力。
- 让急性顾客和多订单调度有意义。

## 成功与奖励规则

### 完成订单

触发：

- 订单气泡 READY 后，点击 READY 订单气泡或当前顾客。

结果：

- 商品交付。
- 顾客开心。
- 收银台入账。
- 顶部收入增加。
- 订单计入完成数。
- 顾客队列推进。

### 完美服务

触发建议：

- 无错误。
- 顾客剩余耐心高于 60%。
- 订单完整交付。

结果建议：

- 额外小费。
- 更高评价。
- 连击继续。
- 后续可显示 PERFECT 或星星反馈。

P0 可以不做独立 PERFECT 大 UI，但数值和反馈规则要预留。

### 连击

P0 规则：

- 连续完成订单增加小费。
- 错误、超时、缺货离开会中断。

设计目的：

- 奖励稳定操作。
- 让熟练玩家和普通玩家产生收入差距。

## 一轮营业压力来源

经营页不能只靠倒计时制造紧张。V1 需要四类压力：

1. 订单识别：看懂顾客要什么。
2. 服务顺序：决定先服务哪位顾客。
3. 加工等待：微波炉占用期间是否处理其他订单。
4. 库存风险：缺货、补货、售罄影响服务。

第 1 天只需要轻量体现这些压力：

- 前两单给商品提示，让玩家理解操作。
- 第三单后取消答案提示。
- 出现一次双顾客竞争同一商品或时间的情况。
- 出现一次需要加热的订单。
- 出现一次库存减少或补货反馈。

## 第 1 天目标体验

第 1 天是试营业，不追求复杂，但必须证明核心玩法成立。

建议体验：

| 阶段 | 时间 | 内容 | 目的 |
| --- | --- | --- | --- |
| 教学暖场 | 0-15 秒 | 单商品订单，商品卡有提示 | 理解取货填入订单气泡 |
| 双顾客开始 | 15-35 秒 | 两位顾客同时等待，可切换 | 理解服务顺序 |
| 轻压力 | 35-60 秒 | 双商品订单或急性顾客 | 产生优先级选择 |
| 加工引入 | 60-80 秒 | 热饭团或微波炉订单 | 理解设备调度 |
| 收尾 | 80-90 秒 | 看能否完成目标单数 | 形成结算期待 |

第 1 天目标：

- 完成 9 单达标。
- 完成 11 单优秀。
- 玩家能清楚知道为什么赚到钱或为什么流失顾客。

## 商业化验收标准

经营页 V1 达标时，应满足：

- 新玩家 1 分钟内能完成第一单。
- 玩家能说出订单气泡是“组装订单和交付”的地方。
- 玩家能看懂哪个商品需要微波炉。
- 玩家能理解为什么某次点击是错误。
- 玩家能理解为什么某位顾客离开。
- 玩家能在一轮后说出下一轮想升级什么或补什么货。
- 美术能根据本规格知道哪些状态必须画出来。
- 开发能根据本规格知道哪些状态必须实现。
- 测试能根据本规格写出手工验收脚本。

## 给美术/UX 的要求

美术和 UI 必须围绕以下玩法信息设计：

- 当前服务顾客。
- 每位顾客订单。
- 商品是否已填入订单气泡。
- 商品是否需要加工。
- 微波炉是否可用/加工中/完成。
- 订单气泡是否 empty/partial/ready/blocked/done。
- 成功付款和收入变化。
- 错误、缺货、超时。

经营页目标图必须服务这些信息。如果某个装饰会干扰玩家理解订单、READY 交付或当前服务对象，应降低优先级。

## 给开发/测试的要求

开发不应先扩展新系统，而应优先保证这些状态稳定：

- `Customer`：waiting、urgent、happy、angry、leaving。
- `OrderItem`：missing、filled、processing、ready_from_device、delivered、error。
- `OrderBubble`：empty、partial、ready、blocked、done。
- `ServiceLock`：none、customer_locked、microwave_locked、clearing。
- `Microwave`：idle、heating、ready、locked。
- `Stock`：available、restocking、sold_out。
- `Payment`：pending、paid、partial、missed。

测试应至少覆盖：

- 普通商品完整交付。
- 多商品订单气泡 partial 到 ready。
- 热食进入微波炉，再进入订单气泡。
- 订单锁定后尝试混单。
- 点错商品。
- 缺货或售罄。
- 顾客超时。
- 完美服务和普通服务收入差异。

## 不属于 V1 的内容

为了控制范围，经营页 V1 不做：

- 三名顾客同时等待。
- 独立交付托盘。
- 多备餐位。
- 提前备餐位。
- 冰柜。
- 拖拽操作。
- 扫码/二次收银。
- 复杂特殊顾客。
- 装修系统。
- 广告或付费。

这些内容可以作为后续商业化扩展，但不能干扰当前经营页打磨。
