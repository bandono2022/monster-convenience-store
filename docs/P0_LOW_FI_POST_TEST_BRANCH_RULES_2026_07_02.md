# P0 Low-Fi Post-Test Branch Rules

日期：2026-07-02

## Owner / Brief / Review / Record

- Owner：Coordination 负责把 Round 14 三方脑暴收成首测后分支规则；Product 负责玩家行为与扩展触发；Art/UI 负责视觉分诊；Development/testing 负责工程排查顺序。
- Brief：本文件定义首测后看到什么结果该走哪条路，避免用加功能、加特效或乱调参掩盖真实问题。
- Review：Product、Art/UI、Development/testing 已分别完成 Round 14 判断。
- Record：本文件；`docs/P0_LOW_FI_FIRST_TEST_ACCEPTANCE_CARD_2026_07_02.md`；`docs/LOCAL_TASK_BOARD.md`。

## Branch 1: Pass To Minimal Implementation Prep

可以进入最小实现准备，必须同时满足：

1. 第 2 局出现服务顺序语言，例如“先热饭团 / 先腾炉 / 先收 READY / 先做右边”。
2. 玩家主动腾微波炉，ready 后取出熟热食释放设备。
3. 玩家主动用备餐位，用于腾炉或下一步计划，不只是误放区。
4. 玩家能区分订单 READY 和微波炉 ready。
5. 误放后能立刻纠正，并不把合法 / 非法 drop 描述成 bug。
6. 玩家愿意再来一局，并归因到“我能更快 / 更顺”。

只有完成率高，但没有上述行为，不算通过。

## Branch 2: Rework Input First

先查输入命中区、drag ghost、吸附、弹回，不调数值：

- 玩家知道要做什么，但拖不到、放不准、被手指挡住。
- 玩家 hover 在正确目标附近，但看不到 target feedback。
- 错误 drop 集中在正确目标附近。
- 玩家说“我想放这里但没吃到”“我看得到但不好拖”“手指挡住了”。
- 第 2 局更快但仍频繁错放。
- 放宽数值后仍烦。

先动：

1. Drop 区。
2. Drag ghost 偏移 / scale / shadow。
3. 吸附半径。
4. Source dim。
5. 非法弹回速度。

不先改玩法、不加功能。

## Branch 3: Rework State / Drop Rules First

先查状态机、drop 规则、slot 占用，不改美术：

- 同一 item 有时能放、有时不能放，玩家复现同操作结果不同。
- 微波炉 ready 取走后仍像被占，或 ready 未取却能放新热食。
- 备餐位显示空但放不进去，或显示占用却被覆盖。
- READY 自动成交、耐心流失、低奖励反馈时序互相冲突。
- Slot 占用、微波炉释放、订单 ready-window 有跳变或不同步。

这类是系统可信度问题。不要用特效遮住规则 bug。

## Branch 4: Rework Visual Cue First

先查视觉 cue，不改规则、不调数值：

- 玩家不知道谁要什么、哪里能放、生热食为什么不能交。
- 玩家混淆订单 READY 和微波炉 ready。
- 玩家不知道微波炉 ready 正在占机，并认为设备坏了。
- 玩家错过 READY countdown 或 red patience danger。
- READY 超时被误读成顾客流失或隐藏惩罚。
- 玩家说“我没看出来”“我以为这里能放”“我以为已经完成了”。

Art/UI 排查顺序：

1. 先修 meaning：这个 cue 到底表达什么。
2. 再修 placement：cue 是否在正确对象上。
3. 再修 strength：颜色、运动、对比是否够。
4. 最后才考虑第二 cue 或更强动画。

允许加强 cue 的条件：

- 玩家能口头理解规则，但运动中漏看状态。
- Cue 位置正确，只是强度太弱。
- 同屏仍少于两个强高亮。
- 新 cue 强化同一位置和同一含义。

禁止加 cue 的条件：

- 玩家不理解规则本身。
- READY 和微波炉 ready 已经混淆。
- 备餐位被误读。
- 根因是手指遮挡或命中区。
- 同屏已有两个强高亮。

## Branch 5: Rework Prep Slot Meaning / Position

先改备餐位位置或造型：

- 玩家称它为库存、商品卡、订单槽、收银 / 支付、按钮。
- 玩家只误放进去，不主动计划使用。
- 玩家期待它接收多件 item 或显示数量。
- 玩家把商品 grid 拖到它当作补货。
- 玩家因它太安静而完全看不到，或因它太亮而误以为是主按钮。

先修：

1. 小托盘语义。
2. 工作台带位置，靠近微波炉但不贴设备。
3. Empty 状态安静。
4. Occupied 只显示一个 item。
5. 只在 valid drag 时高亮。

## Branch 6: Parameter Tuning

只有输入可信、状态机稳定、视觉 cue 可读之后，才允许调参数。

触发信号：

- 玩家看得懂、拖得动，但一局太闲，没有服务顺序判断。
- 玩家不用备餐位也能轻松过。
- 微波炉 ready 占机没有造成取舍。
- READY 窗口太短导致冤，或太长导致无压力。
- 耐心太短，玩家还没形成策略就失败。

首批只允许调：

1. 顾客耐心。
2. 微波炉加工时间。
3. READY 窗口时长。
4. 订单出现节奏。
5. 吸附半径。
6. Ghost 偏移。
7. 误放反馈时长。
8. 小费 / 连击惩罚强度。

不能用参数调优作为理由新增系统。

## Branch 7: Expansion Trigger

扩展必须回应玩家行为里的真实痛点。

允许测试微波炉升级：

- 玩家已经主动腾炉。
- 玩家明确说“炉子太慢 / 堵住了 / 要是快点就好了”。
- 失败或低分主要来自热食等待和设备占机。
- 输入和视觉没有主要问题。

允许测试第二备餐位：

- 玩家已经主动使用 1 个备餐位。
- 玩家多次表达“我还想多放一件”“如果多一格能更顺”。
- 当前失败不是看不懂或拖不准，而是明确缺少暂存空间。
- 微波炉占机和备餐位价值已经被理解。

禁止先加：

- 玩家还不会用 1 个备餐位时，不加第二备餐位。
- 玩家还看不懂状态链时，不加升级。
- 输入、状态机、视觉 cue 未过关时，不加急客、库存、补货、多设备或更多商品。

## Current Verdict

首测后先分诊，再行动。默认顺序是：

`输入可信 -> 状态机稳定 -> 视觉可读 -> 参数调优 -> 扩展测试`

任何一步不通过，都不进入下一步。
