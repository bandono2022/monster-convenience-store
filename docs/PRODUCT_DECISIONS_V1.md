## 决策 1：营业中 HUD 金币口径

营业中 HUD 的主数值显示 `currentShiftRevenue`，即本轮已完成订单带来的营业额。

口径：

- `currentShiftRevenue` 从本轮营业开始时归零。
- 只有订单气泡 READY 后成功交付，才增加 `currentShiftRevenue`。
- 点击商品、微波炉完成、订单气泡 partial、未 READY 点击，都不增加 `currentShiftRevenue`。
- Day 1/3/7 的达标营业额和优秀营业额都使用 `currentShiftRevenue` 判断。
- 收银台付款动效、金币飞行、`+金币` 字样都指向 `currentShiftRevenue` 的增加。

`wallet` 是玩家可花金币，不等于本轮营业额。

口径：

- `wallet` 主要出现在经营准备、采购、升级、结算页。
- 营业中 P0 不强制展示 `wallet`；如果保留，只能作为小号次级信息，不能抢主 HUD。
- 营业中每单入账不直接把 HUD 主数值表现成 `wallet` 增加。
- 结算页再把本轮营业结果结转到 `wallet`；具体按现有经济结算实现，不影响 D-004 验收。

D-004 验收口径：

- 成功交付后，玩家看到的是本轮营业额上涨。
- 测试不再用“钱包是否即时上涨”判断营业内入账是否正确。

D-002 最小实现口径：

- 配置和显示字段要区分 `currentShiftRevenue` 与 `wallet`。
- 不为此新增复杂账本系统。

## 决策 2：订单气泡 READY 交付验收口径

D-004 的核心验收是：订单气泡承担备餐状态和 READY 交付入口，收银台只做入账反馈。

必须通过：

1. 普通商品点击后进入当前顾客订单气泡，不直接收钱。
2. 单品订单填满后，当前顾客订单气泡进入 READY。
3. 多商品订单只填一件时为 partial，能看出已填和缺失项。
4. 未 READY 时点击订单气泡或当前顾客，不完成订单、不加营业额，只给缺失或错误反馈。
5. READY 后点击当前顾客订单气泡，完成交付、顾客开心、订单 DONE、收银台入账、`currentShiftRevenue` 增加。
6. READY 后点击当前顾客区域，也可以触发同一套交付逻辑。
7. 热食先进入微波炉，订单项显示 processing。
8. 微波炉 READY 后必须点击微波炉，成品回到所属顾客订单气泡。
9. 热食回填后如果订单满足，订单气泡进入 READY。
10. 热食不能被切换顾客后填给错误顾客。
11. 错误商品、售罄、补货中都不能误完成订单。
12. 收银台没有独立交付点击职责。

P0 当前顾客规则：

- 只有当前选中顾客的 READY 订单可交付。
- 非当前顾客的 READY 气泡如可点击，P0 只允许切换为当前顾客或提示先服务该顾客，不在同一次点击里交付。

不属于 D-004 必验：

- 三件订单。
- 三名顾客同屏。
- 多备餐位。
- 多微波炉。
- 收银台二次确认或扫码玩法。

## 决策 3：Day 1/3/7 数值落地边界

D-002 只做前 7 天垂直切片的最小可玩验证，先落 Day 1、Day 3、Day 7 三个检查点。

### Day 1

目标：验证普通商品的订单气泡 READY 交付。

边界：

- 只生成普通商品订单。
- 前 2 单可脚本为薯片、柠檬汽水单品。
- 不生成热食、急性顾客、大单客。
- 不把售罄作为新手失败来源。
- 达标口径：8 单或 95 营业额。
- 优秀口径：10 单或 120 营业额。

### Day 3

目标：验证热食、微波炉等待、热食回到所属订单气泡。

边界：

- 首个热食订单固定为 1 件热饭团。
- 首个热食订单不叠双商品、不叠急性顾客。
- 微波炉 Lv.1 为单处理位，加热约 3.4 秒。
- 热食完成后必须回填订单气泡，再 READY 交付。
- 饭团库存保护到首个热食教学完成。
- 达标口径：8 单或 150 营业额。
- 优秀口径：10 单或 190 营业额。

### Day 7

目标：验证综合压力日，但不扩系统。

边界：

- 同屏仍最多 2 位顾客。
- 订单仍最多 2 件。
- 可以混合普通单品、普通双品、热食、普通 + 热食、高价双品。
- 可以出现普通顾客、急性顾客、大单客。
- 每 5 单至少保留 1 单普通单品缓冲。
- 不做 3 件订单、三顾客、多设备、多备餐位。
- 达标口径：10 单或 290 营业额。
- 优秀口径：12 单或 360 营业额。

D-002 不做：

- 30 天经济。
- 通胀、贷款、广告、IAP、离线收益。
- 精确需求预测。
- 新的经营 director 大系统。
- 装修、主题、多店铺。
- 布丁加工规则。

### 决策 4：经营资金与净利润口径

P0 采用现金流口径，不在结算时重复扣成本。

口径：

- 经营准备页采购商品时，`wallet` 已经扣除进货成本。
- 营业中 HUD 主值仍然只显示 `currentShiftRevenue`。
- 结算页显示 `营业额`、`进货成本 / 已消耗商品成本`、`净利润` 三个复盘字段。
- 结算时 `wallet` 增加本轮 `currentShiftRevenue`，因为采购成本已在营业前扣过。
- `净利润 = currentShiftRevenue - goodsCost` 用于结算复盘、长期利润、评级判断和调数值，不作为本轮结算再次加到 `wallet` 的金额。
- 若未来改成“营业前只预留库存、不立即付款”的会计口径，才可以把结算改为 `wallet += netProfit`；该改动必须同步重构采购扣款时机。

验收：

- 一轮完整现金变化应等于：营业前采购扣款 + 营业后收入入账。
- 不允许在当前采购已扣款实现下，把结算改成 `wallet += netProfit`，否则会重复扣除已消耗商品成本。
- 文档里“净利润进入经营资金”的长期表述，在 P0 中解释为“整轮现金结果由采购扣款和营业额入账共同体现”。

### 决策 5：顶部 HUD 第三位

P0 顶部第三 HUD 位定为“服务星级 / 连击质量”槽，不再同时承担满意度、口碑、顾客耐心或热门预测。

口径：

- 左侧主值：本轮营业额 `currentShiftRevenue`。
- 中央主值：剩余时间。
- 右侧主值：服务星级 / 连击质量，用星星、倍率、进度条或简短数字表达。
- 顾客耐心只贴近对应顾客或订单气泡，不进入顶部第三 HUD。
- 店铺口碑主要出现在经营中心、结算和策略选择，不在营业中抢顶部第三位。
- 当前实现中的“满意度百分比”可作为临时工程显示，但 final HUD 文案和资源应转向目标图的星级 / 连击语义。

验收：

- 玩家第一眼能把顶部第三位理解成“本轮服务表现”，而不是玩家生命值、顾客耐心、钱包或热门趋势。
- HUD v2 / Nav 候选不得烘焙动态数字、`x3`、倒计时或进度长度。

### 决策 6：商品卡状态语义

商品卡 P0 只表达当前点击决策需要的信息，不承担经营中心的预测解释。

优先级：

1. 当前服务顾客需要：最强高亮边框 / 发光。
2. 需要加热：火焰角标，只用于饭团等热食流程。
3. 库存：`xN` 小徽章，必须常驻可读。
4. 售罄 / 补货中 / 未解锁：覆盖或压暗状态，不能误读为可点。
5. 热门 / 趋势：营业中 P0 不做强提示；如保留，只能弱化，且不能使用和“当前需要”相同的星星语义。

边界：

- 非当前顾客需要的商品，不在商品卡上做强高亮，避免玩家误以为可以跨单投放。
- 商品名可以降权；商品图标、库存和当前需要状态优先。
- 星星优先保留给顶部服务星级 / 连击，不用来表示商品是否当前需要。

### 决策 7：Day 1 / Day 3 / Day 7 验收边界

P0 只把 Day 1、Day 3、Day 7 作为可执行验收检查点。

口径：

- Day 2 / 4 / 5 / 6 仍是节奏参考，不作为当前 Cocos 手点和自动截图必过项。
- Day 1 验证普通商品、订单气泡 READY、收银入账。
- Day 3 验证首个热饭团、单微波炉等待、热食回到所属订单气泡。
- Day 7 验证 2 顾客、最多 2 件订单、热食 + 普通商品 + 急性 / 大单压力混合。
- Day 3 验收必须能稳定进入“饭团已解锁 + 微波炉 Lv.1 + 饭团有库存”的测试状态；如果自然经营路径暂时不能保证，QA 可以使用脚本 / 预置经济状态验证机制闭环。

暂不做：

- 三件订单、三名顾客、多备餐位、多微波炉、冰柜、拖拽摆放。
- 为下一位顾客提前备餐的独立队列。
- 布丁加工、多个热食加工链、复杂 director 系统。

### 决策 8：P0 熟练空间

P0 的熟练空间来自“选择服务顺序”和“微波炉等待期间处理另一位顾客”，不是提前建立多份备餐队列。

口径：

- 玩家可以切换服务对象。
- 加热中可以处理另一位顾客的简单订单。
- 微波炉成品仍绑定原顾客，不能因为切换顾客而混单。
- 已开始准备的一笔订单必须按服务锁完成、失败或清理。

验收：

- 熟练玩家能通过优先级判断提高收入或连击。
- 页面仍然一眼像目标图，不因加入额外队列或新面板而偏离五段结构。
