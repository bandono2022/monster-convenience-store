# 怪兽便利店美术规范

更新时间：2026-06-27

## 唯一权威目标图

当前 gameplay 页面唯一权威目标图是：

`assets/ui/mockups/gameplay-main-order-bubble-ready-v2.png`

所有 final art 候选、整屏 fidelity 判断、布局优先级和资源质量验收，都以这张图为准。

`assets/reference/style-exploration-flat-cartoon-v1.png` 只作为风格参考，不是目标图。旧 mockup、旧 P0 资源、旧 READY 临时方案只能作为 archived / reference，不得反向约束当前 final art。

## 当前玩家理解目标

美术不是只负责“更好看”，必须帮助玩家在手机竖屏中快速看懂经营链路：

- 当前正在服务哪位顾客。
- 订单气泡里还缺什么、已填什么、是否 READY。
- 商品区可以点击，热食需要微波炉。
- READY 订单气泡是交付入口。
- 收银机和 HUD 只表达成功后的入账反馈，不是独立交付按钮。
- `390 x 844` 下 READY、缺失 / 已填、库存、热食、微波炉三态仍清楚可读。

## 过时内容

以下旧口径已过时：

- “只以 flat cartoon 探索图定义最终画面”过时；它现在只是 style reference。
- “非 READY 区域单独收敛”过时；当前按整屏商业化质量验收。
- “修补、拆分、盖字当前 baked READY 占位”过时；READY 属于 final art 统一替换。
- “P0 占位可继续调成 final”过时；P0 只可参考接入结构和尺寸经验。
- “背景、顾客、设备、HUD / dock 后置不管”过时；它们属于整屏 final art 挑战。
- “独立交付托盘 / DELIVERED 按钮 / 传送带式交付”过时；当前目标是订单气泡 READY 交付。

## 核心风格规则

- 商业休闲手游品质，明亮、友善、圆润、厚描边。
- 视角以目标图为准：正面略带俯视，手机竖屏 `750 x 1334`。
- 主轮廓使用深紫黑，不用纯黑。
- 色块清晰，高饱和但不脏；避免写实、厚涂、3D 黏土、像素风。
- 光源统一来自左上方。
- 手机实际尺寸下优先可读性，细节服从轮廓和功能识别。
- 图片内禁止无关文字、品牌、logo、水印、版权角色。

## 角色规范

- 当前顾客：友好、开心、可交付，柜台前半身清晰，手可作为独立 overlay。
- 等待顾客：同风格同尺度，表情可紧张或不满，但保持可爱。
- 怪兽眼睛、嘴、手部动作必须在 `390 x 844` 下可读。
- 不做复杂毛发、细碎纹理或无法拆分的整屏角色截图。
- 当前顾客的突出感来自手搭柜台、READY 气泡、光效和遮挡关系，不来自基础体型变大。

## 订单气泡规范

- 奶油白底、深紫黑粗描边、圆角大形状，尾巴指向对应顾客。
- 当前订单气泡更突出，可有黄色 / 绿色完成高亮。
- 等待订单气泡更弱，但商品需求必须清楚。
- 槽位、已填态、勾选、错误态、READY 状态应可分层替换。
- 不把商品 icon、数量、动态文案烘焙进气泡底板。
- 订单槽在 `390 x 844` 下视觉可辨高度不低于 44px。

## READY 资源规范

- 当前 runtime `ready_badge_compact.png` 是 frozen baked 占位，只能继续冻结，不能拆图、盖字、补字或局部修。
- final READY 应包含：胶囊底板、发光、完成勾 / 槽位状态，可由 runtime 动态叠加 READY 文案。
- READY 必须在 `390 x 844` 下可读，并和订单气泡形成明确完成反馈。
- 呼吸、缩放、闪光优先 runtime 动画，不生成整段动效序列。

## 商品卡与商品规范

- 商品卡：奶油底、深紫描边、圆角、清楚库存区域、需要态 glow。
- 商品 icon：透明背景、方形构图、主体约 80%，订单气泡内也能识别。
- 首批商品：饭团、怪兽薯片、柠檬饮料、草莓牛奶、布丁、星星糖。
- 库存数字、价格、商品名、售罄字样不得烘焙进图片。
- 商品卡候选必须能在当前 runtime 约 `205 x 180` 的卡片尺寸下读清，否则属于布局改造，不是单纯资源替换。

## 设备与工作台规范

- 微波炉三态必须不用读字也能区分：idle / heating / ready。
- 收银机 idle / pay 必须清楚表达付款与收益反馈。
- 收银机资产只能强化“已完成后入账”，不能暗示扫码、二次确认或独立交付。
- 柜台前景服务触摸与物件摆放，不遮挡核心操作。
- 进度条、倒计时、收益数字、粒子动画优先 runtime 绘制或叠加。

## 背景规范

- 便利店背景需有货架、灯光、空间深度，但不能抢订单、顾客、商品卡。
- 中央交互区域保持干净，前景 / 中景 / 背景层级清晰。
- 背景可以作为 final candidate，但不得把整张目标图截图直接当 runtime 背景。

## 生成与验收流程

1. 生成前先确认 owner：美术 / UI 提供 brief、prompt 方向、输出目录和视觉验收标准。
2. 候选资源先进入 `assets/ui/final-candidates/`，不直接覆盖 runtime。
3. 每批先生成少量候选，由美术 / UI 选样，再让产品和开发复核。
4. 产品复核玩家是否看懂当前顾客、订单缺失、READY 交付和收银反馈。
5. 开发 / 测试复核透明边界、九宫格、尺寸、`390 x 844` / `750 x 1334` 适配和 touch 风险。
6. 最终接入前才复制到 `assets/resources/`，并保留旧资源 fallback。

## 永不作为 Final Art 候选

- 程序绘制的占位 PNG 冒充 final art。
- 目标图截图、mockup 切片、整区截图按钮。
- 含 baked 数字、库存、倒计时、收入、READY / DONE / 售罄文字的底图。
- 含 logo、水印、真实品牌、版权角色、来源不明素材。
- 3D 黏土、写实、像素、厚涂、暗黑恐怖风。
- 当前 frozen `ready_badge_compact.png` 的拆分、盖字、二次修补版本。

## 禁止扩展暗示

第一批 final art 不应暗示以下未做功能：

- 三顾客、三件以上订单。
- 多微波炉、多备餐位、提前备餐、拖拽玩法。
- 独立交付托盘、DELIVERED 按钮、扫码或二次收银。
- 30 天经济、装修、广告 / IAP、复杂特殊顾客。
